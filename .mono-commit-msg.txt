Production-readiness Section B: DB integrity constraints + concurrency-safe writes + N+1 fix (ASCII-only)

DB INTEGRITY (new lib/dbHardening, run once at boot): dedupe + add unique indexes that the app-level check-then-insert couldn't guarantee under concurrency (same learner runs on web + WhatsApp) - one gradebook entry per (assignment,user), one VALID credential per (user,module) [partial index], one funded-seat per (agreement,learner). Never throws; skips absent tables.

CONCURRENCY-SAFE WRITES now backed by those indexes: applyGrade is wrapped in a db.transaction (submission grade + gradebook mirror + notification are atomic - no more 'score written but entry missing' window) and mirrors via a real onConflictDoUpdate. Credential issuance uses onConflictDoNothing on the partial index (two concurrent checkpoints can't double-issue). Funded-seat assignment locks the agreement row FOR UPDATE, re-checks the cap inside the transaction, and inserts onConflictDoNothing (no more seat over-allocation / double-count).

PERF: /coach/learners was 2 DB queries PER learner (unbounded N+1, whole platform for a super_admin) - batched into two grouped queries (credential counts + last session).

VALIDATION: esbuild clean on dbHardening.ts, index.ts, assignments.ts, funder.ts, mastery.ts, coach.ts.
