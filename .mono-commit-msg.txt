Learning Hub persistence B: DB tables + Supabase Storage upload + CRUD API (additive) (ASCII-only)

MIGRATION REQUIRED: run praxis/migration-learning-hub.sql against the Praxis Postgres (creates learning_content, course_templates, course_assignments + seeds). Additive/idempotent.

Backend only, no frontend behaviour change yet (UI still uses in-session state until the C increment rewires it, so this deploy is safe).

- New Drizzle tables in lib/db schema/learning_hub.ts (exported from schema index).
- New api-server route learningHub.ts mounted in routes/index: GET/POST content, POST content/upload (forwards file to Supabase Storage via REST), PATCH review, DELETE, templates GET/POST, assignments GET/PUT/toggle. Writes gated to super_admin.
- src/lib/supabaseStorage.ts (REST, no SDK): reads SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY at call time; storageEnabled() so missing config fails cleanly.

ENV (for uploads): set SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY on the Praxis service and create a Storage bucket (default name learning-hub).

VALIDATION: esbuild syntax-check clean on supabaseStorage, learningHub route, schema, routes index.
