Perf: cache gradebook structure (columns + settings) with TTL + write invalidation

The gradebook re-ran the same structural queries (which columns exist, weighting/letter bands) on every matrix load and every grade-write alert recompute. Now getCourseColumns (keyed by course+org) and getGradebookSettings are cached in-process with a short TTL (columns 15s, settings 30s) and explicitly invalidated on every structural write: settings PUT, sync, gradebook-items POST/PATCH/DELETE, config PUT (course + org override), and the cell handler's auto-created assignment override. Scores and cells are NEVER cached (computeLearner + getScoreData still read fresh), so grades can never drift; only the stable structure is memoised. Per-instance cache, short TTL bounds any staleness. 17 engine unit tests still green. Files: lib/gradebookEngine.ts, routes/gradebook.ts.
