Section B perf: SQL-count the platform alerts + bound roster queries (ASCII-only)

/platform/alerts pulled SIX whole tables (invoices, documents, partners, courses, users, enrolments) into Node and filtered with .length - full-table scans that grow linearly with platform size (users/enrolments worst). Replaced with SQL count(*) WHERE. Roster reads (/partners/:id/members, /organisations/:id/members) were unbounded - added a LIMIT 2000 bound (cursor pagination is a follow-up) so a very large partner can't return an unbounded result set.
